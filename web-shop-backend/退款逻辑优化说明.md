# 退款逻辑优化说明

## 📋 修改时间
2025-11-06

## 🎯 修改目的
优化退款申请逻辑，允许已支付但未发货（status=1）的订单也能申请退款，并优化退款审核和取消后的订单状态恢复逻辑。

---

## 🔧 修改内容

### 1️⃣ 提交退款申请 - `applyRefund()`

#### ❌ 修改前
```java
// 2. 验证订单状态（只有已支付或已发货的订单才能退款）
if (order.getStatus() != 2 && order.getStatus() != 3) {
    throw new IllegalArgumentException("订单状态不支持退款");
}
```

**问题**：只允许 status=2 和 status=3 的订单退款，但 status=1（已支付但未发货）的订单也应该能退款。

#### ✅ 修改后
```java
// 2. 验证订单状态（只有已支付、已发货、已收货的订单才能退款）
// status: 1-待发货(已支付), 2-已发货, 3-已收货
if (order.getStatus() != 1 && order.getStatus() != 2 && order.getStatus() != 3) {
    throw new IllegalArgumentException("订单状态不支持退款，只有已支付、已发货、已收货的订单可以退款");
}
```

**改进**：
- ✅ 允许 status=1（待发货/已支付）的订单申请退款
- ✅ 更详细的错误提示信息
- ✅ 添加状态说明注释

---

### 2️⃣ 审核退款申请 - `reviewRefund()`

#### ❌ 修改前
```java
// 4. 更新订单状态
Orders order = ordersService.getById(refund.getOrderId());
if (status == 1) {
    // 同意退款
    order.setStatus(7); // 已退款
    log.info("退款申请已同意");
} else {
    // 拒绝退款，恢复原状态
    order.setStatus(2); // 恢复为已支付  ❌ 硬编码
    log.info("退款申请已拒绝");
}
```

**问题**：拒绝退款后，所有订单都恢复为 status=2，不考虑原始状态。

#### ✅ 修改后
```java
// 3. 获取订单信息（用于记录原始状态）
Orders order = ordersService.getById(refund.getOrderId());
if (order == null) {
    throw new IllegalArgumentException("关联订单不存在");
}

// 4. 更新退款状态
refund.setStatus(status);
refund.setRemark(remark);
this.updateById(refund);

// 5. 更新订单状态
if (status == 1) {
    // 同意退款
    order.setStatus(7); // 已退款
    log.info("退款申请已同意，订单状态更新为已退款");
} else {
    // 拒绝退款，需要根据支付时间和发货时间判断应该恢复到什么状态
    if (order.getShipTime() != null) {
        // 已发货，恢复为已发货状态
        order.setStatus(2);
        log.info("退款申请已拒绝，订单状态恢复为已发货");
    } else {
        // 未发货，恢复为待发货状态
        order.setStatus(1);
        log.info("退款申请已拒绝，订单状态恢复为待发货");
    }
}
```

**改进**：
- ✅ 智能判断订单原始状态
- ✅ 根据 `shipTime` 判断应恢复为 status=1 还是 status=2
- ✅ 更详细的日志记录
- ✅ 增加订单存在性验证

---

### 3️⃣ 取消退款申请 - `cancelRefund()`

#### ❌ 修改前
```java
// 5. 恢复订单状态
order.setStatus(2); // 恢复为已支付  ❌ 硬编码
ordersService.updateById(order);
```

**问题**：取消退款后，所有订单都恢复为 status=2，不考虑实际状态。

#### ✅ 修改后
```java
// 5. 恢复订单状态，根据发货情况判断
if (order.getReceiveTime() != null) {
    // 已收货
    order.setStatus(3);
    log.info("退款申请已取消，订单状态恢复为已收货");
} else if (order.getShipTime() != null) {
    // 已发货
    order.setStatus(2);
    log.info("退款申请已取消，订单状态恢复为已发货");
} else {
    // 未发货
    order.setStatus(1);
    log.info("退款申请已取消，订单状态恢复为待发货");
}
```

**改进**：
- ✅ 根据 `receiveTime` 和 `shipTime` 判断应恢复的状态
- ✅ 支持三种状态恢复：待发货(1)、已发货(2)、已收货(3)
- ✅ 更详细的日志记录
- ✅ 增加订单存在性验证

## 📊 订单状态流转图

### 修改前
```
已支付(2) ───┐
             ├──> 退款中(6) ──┬──> 已退款(7)  [审核通过]
已发货(3) ───┘               └──> 已支付(2)  [审核拒绝/取消] ❌ 硬编码
```

### 修改后
```
待发货(1) ───┐
已发货(2) ───┼──> 退款中(6) ──┬──> 已退款(7)     [审核通过]
已收货(3) ───┘               │
                             ├──> 待发货(1)     [拒绝/取消，未发货]
                             ├──> 已发货(2)     [拒绝/取消，已发货]
                             └──> 已收货(3)     [取消，已收货]
```

## 🎯 业务场景示例

### 场景 1：待发货订单退款
```
用户下单 → 支付成功 → status=1 (待发货)
         ↓
用户申请退款 → status=6 (退款中)
         ↓
商家同意 → status=7 (已退款) ✅
商家拒绝 → status=1 (恢复为待发货) ✅
用户取消 → status=1 (恢复为待发货) ✅
```

### 场景 2：已发货订单退款
```
用户下单 → 支付成功 → 商家发货 → status=2 (已发货)
                              ↓
                    用户申请退款 → status=6 (退款中)
                              ↓
                    商家同意 → status=7 (已退款) ✅
                    商家拒绝 → status=2 (恢复为已发货) ✅
                    用户取消 → status=2 (恢复为已发货) ✅
```

### 场景 3：已收货订单退款
```
用户下单 → 支付 → 发货 → 确认收货 → status=3 (已收货)
                                  ↓
                        用户申请退款 → status=6 (退款中)
                                  ↓
                        商家同意 → status=7 (已退款) ✅
                        商家拒绝 → status=2 (恢复为已发货) ✅
                        用户取消 → status=3 (恢复为已收货) ✅
```
