# 修改密码接口优化说明

**修改日期：** 2025-11-04  
**优化内容：** 将修改密码接口从 Query Parameters 改为 Request Body

---

## 🎯 修改原因

### 原有方案的问题

**之前使用 Query Parameters：**
```
PUT /api/user/5/password?oldPassword=123456&newPassword=654321
```

**存在的安全隐患：**
1. ❌ 密码在 URL 中明文显示
2. ❌ 浏览器历史记录会保存完整 URL（包含密码）
3. ❌ 服务器访问日志会记录 URL（密码泄露）
4. ❌ 代理服务器可能缓存 URL
5. ❌ URL 长度有限制

### 优化后的方案

**现在使用 Request Body：**
```
PUT /api/user/5/password
Content-Type: application/json

{
  "oldPassword": "123456",
  "newPassword": "654321",
  "confirmPassword": "654321"
}
```

**优势：**
1. ✅ 密码不在 URL 中显示
2. ✅ 浏览器历史记录不会保存密码
3. ✅ 服务器日志默认不记录请求体
4. ✅ 更符合 RESTful 规范（敏感数据用 Body）
5. ✅ 支持前端二次确认（confirmPassword）
6. ✅ 便于添加参数验证

## 📝 修改内容

### 1. 新建 DTO 类

**文件：** `src/main/java/org/javaweb/webshopbackend/pojo/dto/ChangePasswordDTO.java`

```java
@Data
@Schema(description = "修改密码请求")
public class ChangePasswordDTO {

    @Schema(description = "原密码")
    @NotBlank(message = "原密码不能为空")
    private String oldPassword;

    @Schema(description = "新密码")
    @NotBlank(message = "新密码不能为空")
    @Size(min = 6, max = 20, message = "新密码长度必须在6-20位之间")
    private String newPassword;

    @Schema(description = "确认新密码")
    @NotBlank(message = "确认密码不能为空")
    private String confirmPassword;
}
```

**新增特性：**
- ✅ 使用 `@NotBlank` 验证非空
- ✅ 使用 `@Size` 限制密码长度（6-20位）
- ✅ 新增 `confirmPassword` 字段，支持前端二次确认

---

### 2. 修改 Controller

**文件：** `src/main/java/org/javaweb/webshopbackend/controller/UserController.java`

**修改前：**
```java
@PutMapping("/{userId}/password")
public Result<Void> updatePassword(
        @PathVariable Long userId,
        @RequestParam String oldPassword,  // ← Query Parameter
        @RequestParam String newPassword) {
    
    userService.changePassword(userId, oldPassword, newPassword);
    return Result.success("密码修改成功");
}
```

**修改后：**
```java
@PutMapping("/{userId}/password")
public Result<Void> updatePassword(
        @PathVariable Long userId,
        @Valid @RequestBody ChangePasswordDTO changePasswordDTO) {  // ← Request Body
    
    // 验证新密码和确认密码是否一致
    if (!changePasswordDTO.getNewPassword().equals(changePasswordDTO.getConfirmPassword())) {
        throw new IllegalArgumentException("两次输入的新密码不一致");
    }
    
    userService.changePassword(userId, 
        changePasswordDTO.getOldPassword(), 
        changePasswordDTO.getNewPassword());
    
    return Result.success("密码修改成功");
}
```

**改进点：**
- ✅ 使用 `@RequestBody` 接收请求体
- ✅ 使用 `@Valid` 触发参数校验
- ✅ 添加两次密码一致性验证

## 🔐 安全性对比

### Query Parameters（之前）

| 场景 | 安全性 | 说明 |
|------|--------|------|
| URL 显示 | ❌ 不安全 | 密码在 URL 中明文显示 |
| 浏览器历史 | ❌ 不安全 | 保存完整 URL（含密码） |
| 服务器日志 | ❌ 不安全 | 默认记录完整 URL |
| 代理缓存 | ❌ 不安全 | 可能被代理服务器缓存 |
| HTTPS 保护 | ✅ 安全 | HTTPS 可以加密 URL |
| 日志审计 | ❌ 麻烦 | 日志中密码需脱敏处理 |

### Request Body（现在）

| 场景 | 安全性 | 说明 |
|------|--------|------|
| URL 显示 | ✅ 安全 | 密码不在 URL 中 |
| 浏览器历史 | ✅ 安全 | 只保存 URL，不含密码 |
| 服务器日志 | ✅ 安全 | 默认不记录请求体 |
| 代理缓存 | ✅ 安全 | 请求体通常不被缓存 |
| HTTPS 保护 | ✅ 安全 | HTTPS 加密请求体 |
| 日志审计 | ✅ 方便 | 日志中无密码信息 |

## 🎓 RESTful 最佳实践

### URL vs Body 的选择原则

#### 使用 Query Parameters（URL）的场景：

```java
@GetMapping("/users")
public Result<List<User>> getUsers(
    @RequestParam(required = false) String keyword,  // ✅ 搜索关键词
    @RequestParam(defaultValue = "1") int pageNum,   // ✅ 分页参数
    @RequestParam(defaultValue = "10") int pageSize  // ✅ 分页大小
)
```

**适用于：**
- ✅ 非敏感数据（搜索关键词、排序字段）
- ✅ 过滤条件（状态、分类）
- ✅ 分页参数（pageNum、pageSize）
- ✅ GET 请求（RESTful 规范）
- ✅ 可缓存的请求

---

#### 使用 Request Body 的场景：

```java
@PostMapping("/login")
public Result<User> login(@RequestBody LoginDTO loginDTO)

@PutMapping("/{userId}/password")
public Result<Void> updatePassword(
    @PathVariable Long userId,
    @RequestBody ChangePasswordDTO dto  // ✅ 敏感数据
)

@PostMapping("/register")
public Result<User> register(@RequestBody RegisterDTO dto)  // ✅ 复杂对象
```

**适用于：**
- ✅ 敏感数据（密码、身份证号、银行卡）
- ✅ 复杂对象（多层嵌套数据）
- ✅ 大量参数（3个以上）
- ✅ POST/PUT/PATCH 请求
- ✅ 二进制数据（文件上传）

## ⚠️ 注意事项

### 1. 前端需要相应修改

**之前的前端代码：**
```javascript
// ❌ 旧代码
axios.put(`/api/user/${userId}/password`, null, {
  params: {
    oldPassword: '123456',
    newPassword: '654321'
  }
})
```

**现在的前端代码：**
```javascript
// ✅ 新代码
axios.put(`/api/user/${userId}/password`, {
  oldPassword: '123456',
  newPassword: '654321',
  confirmPassword: '654321'
})
```

### 2. 仍需 HTTPS 保护

**重要提示：**
- Request Body 只是将密码从 URL 移到请求体
- 如果没有 HTTPS，密码仍然是明文传输
- **生产环境必须使用 HTTPS！**

```
┌─────────────┐          HTTP（明文）          ┌─────────────┐
│   浏览器    │ ─────────────────────────────> │   服务器    │
│             │   密码可被中间人截获 ❌         │             │
└─────────────┘                                 └─────────────┘

┌─────────────┐         HTTPS（加密）          ┌─────────────┐
│   浏览器    │ ═══════════════════════════════> │   服务器    │
│             │   密码被加密传输 ✅             │             │
└─────────────┘                                 └─────────────┘
```

### 3. 后端仍需哈希存储

**完整的密码安全方案：**
```
前端                    传输                    后端
  │                      │                        │
  │ 用户输入             │                        │
  │ 明文密码: 123456     │                        │
  │                     │                        │
  ├──── HTTPS 加密 ────>│                        │
  │     (Request Body)  │                        │
  │                     │                        │
  │                     ├──── HTTPS 解密 ───────>│
  │                     │                        │
  │                     │                        │ 收到明文: 123456
  │                     │                        │      ↓
  │                     │                        │ 哈希运算 (BCrypt/MD5)
  │                     │                        │      ↓
  │                     │                        │ 存入数据库: e10adc...
```
